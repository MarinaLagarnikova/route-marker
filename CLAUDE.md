# Вешка — веб-приложение для отметки пройденного маршрута по GPX

## Что это

Мобильное веб-приложение для туристов. Пользователь загружает GPX-трек маршрута
и по мере прохождения вручную отмечает контрольные точки как пройденные.
GPS не используется вовсе: приложение — «умный чек-лист» поверх карты,
рассчитанный на условия, где GPS и интернет ненадёжны.

Рабочее название — «Вешка». Использовать его в UI (заголовок, title),
но не хардкодить по коду — вынести в конфиг `shared/config`.

## Стек

- React 18 + TypeScript + Vite
- Tailwind CSS + shadcn/ui (компоненты добавлять через CLI, не копипастой)
- Архитектура: Feature-Sliced Design (FSD)
- Карта: Яндекс Карты JS API v3 — подключать ТОЛЬКО через адаптер
  (см. раздел «Карта»). Leaflet и тайлы OpenStreetMap НЕ использовать.
- Состояние: zustand (без redux)
- Хранение прогресса: localStorage через обёртку в `shared/lib/storage`

## Структура FSD

```
src/
  app/          — точка входа, роутер, провайдеры, глобальные стили
  pages/
    start/      — стартовый экран
    route/      — основной экран маршрута
  widgets/
    route-header/     — шапка: название маршрута + кнопка «на главную»
    progress-panel/   — блок «пройдено / осталось / прогноз финиша»
    route-map/        — карта с треком и точками
    checkpoint-list/  — список контрольных точек
  features/
    upload-gpx/       — выбор файла, парсинг, валидация, ввод названия
    mark-checkpoint/  — отметка/отмена точки, автоопределение направления
    reset-progress/   — сброс отметок
  entities/
    route/            — модель маршрута: точки трека, дистанции, стор
    checkpoint/       — модель контрольной точки, вычисление статусов
  shared/
    ui/               — shadcn-компоненты
    lib/gpx/          — парсер GPX (без внешних зависимостей, DOMParser)
    lib/geo/          — гаверсинус, кумулятивные дистанции, проекция wpt на трек
    lib/storage/      — типизированная обёртка над localStorage
    config/           — название приложения, ключ API карт (из .env)
```

Правила FSD соблюдать строго: импорты только сверху вниз
(pages → widgets → features → entities → shared), кросс-импорты
между слайсами одного слоя запрещены.

## Экраны и флоу

### 1. Стартовый экран (`pages/start`)

- Кнопка «Загрузить GPX» (input type=file, accept=".gpx").
- Текстовый инпут «Название маршрута». После выбора файла — предзаполнить
  названием из GPX (`<trk><name>` или `<metadata><name>`), пользователь
  может отредактировать.
- Кнопка «Начать» активна, когда файл распарсен без ошибок и название непустое.
- Если в localStorage есть незавершённый маршрут — показать карточку
  «Продолжить: {название}, пройдено X из Y км» с кнопкой продолжения.
- Ошибка парсинга (нет trkpt/rtept, битый XML) — понятное сообщение под
  кнопкой, не alert.

### 2. Основной экран (`pages/route`)

Сверху вниз:

1. **Шапка** (`widgets/route-header`): название маршрута + кнопка «на главную»
   (иконка домика). Переход на главную НЕ стирает прогресс — он в localStorage.
2. **Блок расстояния** (`widgets/progress-panel`): пройдено км, осталось км,
   прогноз времени финиша. ВАЖНО: дизайн этого блока временный, его будут
   переделывать вручную — разметку держать простой и плоской, всю логику
   вычислений вынести в `entities/route` (селекторы стора), чтобы замена
   вёрстки не трогала логику.
3. **Карта** (`widgets/route-map`): трек и контрольные точки, см. «Карта».
4. **Список точек** (`widgets/checkpoint-list`): имя, километраж,
   время отметки; кнопка «Здесь» / «Отменить». Touch-таргеты минимум 44px.

Блоков «направление» и «профиль высот» НЕТ — не добавлять.

## Домённая логика

### Парсинг GPX

- Точки трека: `trkpt` (fallback: `rtept`). lat/lon обязательны, ele игнорировать.
- Контрольные точки: `wpt` с их названиями из `<name>`.
  Каждый wpt проецируется на ближайшую точку трека; точки сортируются
  по позиции вдоль трека.
- Fallback без настроек в UI: если waypoints меньше 3 — автоматически
  расставить точки каждый 1 км с именами «КТ 1», «КТ 2»…
- Старт и финиш трека всегда являются контрольными точками
  («Старт» / «Финиш», либо имя ближайшего wpt с префиксом).

### Направление — автоопределение, без UI

Переключателя направления нет. Правила:

- До первой отметки порядок точек — как в файле.
- Первая отметка задаёт направление: если отмеченная точка находится
  во второй половине трека по километражу (km > total/2), порядок точек,
  километраж и нумерация разворачиваются — эта сторона становится стартом.
- После разворота отметка применяется уже в новой системе координат.
- Направление фиксируется до сброса отметок (`features/reset-progress`,
  кнопка в шапке списка точек, с confirm-диалогом shadcn).

### Правила отметки

- Отметка точки помечает пройденными её и все предыдущие (в текущем
  направлении). У каждой новоотмеченной точки сохраняется timestamp.
- Отменить можно только последнюю пройденную (повторный тап по ней).
  Тап по давно пройденной точке игнорируется.
- Прогноз финиша: считается после ≥2 отметок как
  `время_последней + осталось_км / средний_темп`, где темп =
  пройдено_км / (время_последней − время_первой). Показывать «—»,
  пока данных мало.

### Персистентность

- Весь стейт маршрута (точки, отметки, направление, название) сохраняется
  в localStorage при каждом изменении; ключ — hash содержимого GPX.
- При повторной загрузке того же файла — предложить продолжить с сохранённым
  прогрессом.

## Карта

- Провайдер: Яндекс Карты JS API v3. Ключ — в `.env`
  (`VITE_YANDEX_MAPS_API_KEY`), получить на developer.tech.yandex.ru.
  В README описать, как получить ключ.
- ОБЯЗАТЕЛЬНО: изолировать провайдера за интерфейсом
  `shared/lib/map-adapter` (initMap, drawTrack, drawCheckpoint,
  fitBounds, onCheckpointTap) — чтобы провайдер менялся в одном модуле.
  Компоненты виджета route-map не должны импортировать ymaps напрямую.
- Слои: схема по умолчанию, переключатель на спутник/гибрид.
- Отрисовка: оставшийся трек — пунктирная линия приглушённого цвета,
  пройденный участок — сплошная красная поверх. Пройденные точки — маркер
  «бело-красная марка», непройденные — светлый кружок с тёмной обводкой.
  Тап по маркеру = отметка точки.
- Карта должна переживать отсутствие сети: ошибки загрузки тайлов не должны
  ломать интерфейс, трек и точки рисуются в любом случае.

## Адаптив и мобильность

- Mobile-first: основной таргет — телефон в вертикальной ориентации,
  ширина 360–430px. Десктоп — просто центрированная колонка max-width 560px,
  отдельной десктопной вёрстки не делать.
- Никакого hover-only поведения; все действия доступны тапом.
- viewport meta с user-scalable=no не ставить (доступность), но контролы
  делать достаточно крупными, чтобы зум не понадобился.

## Не делать (пока)

- GPS/geolocation API использовать только как опциональный индикатор текущего положения на карте.
  Если геолокация доступна — показывать метку пользователя на карте.
  Если недоступна или отклонена — приложение работает полностью без неё, никаких ошибок и деградации UX.
- Никакой авторизации, бэкенда, аккаунтов — приложение полностью клиентское.
- PWA/service worker — следующий этап, каркас не закладывать, но и не блокировать.
- Профиль высот, настройка интервала точек, переключатель направления —
  осознанно исключены, не возвращать.

## Качество

- TypeScript strict, без any в домённой логике.
- Юнит-тесты (vitest) на: парсер GPX, вычисление дистанций, проекцию wpt
  на трек, автоопределение направления, правила отметки/отмены.
- Логика — в entities/features, компоненты тонкие.
- Все тексты интерфейса — по-русски, вынесены в константы слайса.
